| 2026-06-18 | Die Best-of-Satz-Zählung (Duell-Siege je Seite) war in vier Anzeige-Flächen reimplementiert (Standings, Spielplan-Label, PDF, Erfassungs-Dialog). Als das Stechschuss-Modell kippte (per Stechschuss entschiedenes Gleichstands-Duell zählt jetzt mit → 2:1 statt 1:1), musste die Logik an allen vier Stellen angefasst werden. | Eine Domänen-Berechnung wie die Satz-Zählung von Anfang an in EINER reinen Funktion kapseln (`bestOfDuelTally` in `bestOf.ts`) und überall daraus konsumieren — nicht je Anzeige-Fläche neu zählen. |
| 2026-06-18 | Der Stechschuss war spec'd + Sportleitungs-freigegeben als reiner Match-Entscheid (Satz bleibt „1:1"). Am echten Test-Datensatz empfand der User das als unintuitiv und kippte die Entscheidung auf „Stechschuss-Duell zählt → 2:1". | Wertungs-/Domänenentscheidungen am echten Datensatz mit dem Domänen-Owner gegenprüfen, bevor sie als endgültig gelten — besonders wenn sie kontraintuitiv wirken könnten. Spec-Entscheidungen sind revidierbar; die Modell-Logik dafür zentral halten. |
| 2026-06-18 | Den Series-Unique-Index von `(matchupId, participantId)` auf `(matchupId, participantId, duelNumber)` umgestellt — aber bestehende/klassische Serien hätten `duelNumber = NULL`. SQL behandelt NULLs als distinct, d.h. der Index hätte die Eindeutigkeit klassischer Serien still verloren (mehrere NULL-Zeilen pro Schütze/Matchup möglich). Vom Spec-Review gefangen | Beim Erweitern eines Unique-Index um eine **nullable** Spalte: bestehende NULL-Zeilen auf einen Nicht-NULL-Sentinel backfillen (hier `duelNumber=1` für klassische Serien) **und** den Neuschreiber (`saveMatchResult`) den Sentinel setzen lassen — sonst greift die Eindeutigkeit nicht |
| 2026-06-18 | Der Stechschuss (Einzelschuss-Stichentscheid) wurde anfangs über `duelOutcome(scoringMode)` aufgelöst. Für RINGTEILER/RINGS ging das per Storage-Trick (`ringteiler = maxRings − Schuss`), aber im TEILER-Modus (Stechschuss-Teiler = 0) lieferte es immer TIE → der Stechschuss löste sich nie auf | Ein Einzelschuss-Tiebreak wird **immer über sein eigenes Maß** (Schusswert, höher gewinnt) entschieden, nicht über den league-`scoringMode` — eigene Funktion `stechschussOutcome(homeShot, awayShot)`, in Action **und** Standings **und** PDF konsistent nutzen |
| 2026-06-18 | Feature (Best-of-Liga) und paralleler Bugfix (Teiler-Faktor nur bei gemischt) liefen gleichzeitig; beide berühren die Ringteiler-/Faktor-Berechnung. Auf dem alten Stand fertiggebaut wären `duelOutcome`/Standings mit roher statt effektiver Faktor-Logik entstanden → Rework | Bei parallelen Strängen im selben Bereich: erst auf den gelandeten Fix **rebasen**, dann die eigenen Annahmen gegen die korrigierte Logik prüfen, dann weiterbauen. Hier: `correctedTeiler = teiler × effectiveTeilerFaktor(...)` von Anfang an mitgedacht |
